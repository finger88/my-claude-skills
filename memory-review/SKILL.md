# memory-review

## 目的

把积压在 `_pending.md` 里的候选记忆，经用户确认后正式写入 MEMORY.md。
同时维护 MEMORY.md 的健康：衰减过期条目，识别可以毕业的稳定模式。

## 触发方式

- 用户说"复盘"、"memory-review"、"整理一下记忆"等
- 每周五/周日启动时，AI 主动提示是否进行复盘
- 不自动执行，必须用户确认后才开始

---

## 执行流程

### 第一步：处理 _pending.md

1. 读取 `00 专注区/_pending.md` 全部待处理条目
2. 逐条向用户展示，询问：**记 / 不记 / 改一下**
3. 用户确认后：
   - `记` → 写入 MEMORY.md 对应区块（L2 动态 或 L3 程序性），带元数据
   - `不记` → 从 _pending.md 删除
   - `改一下` → 用户修改描述后再写入
4. 全部处理完后，清空 `_pending.md` 的「待处理条目」区块

### 第二步：检查衰减

1. 读取 MEMORY.md 所有条目的 `last_activated` 字段
2. 计算距今超过 4 周未被引用的条目
3. 向用户报告：「这些条目超过 4 周未引用，建议降级或清除」
4. 用户决定：**降级 / 保留 / 清除**

### 第三步：识别毕业候选

1. 扫描 MEMORY.md 中强度为 ★★★ 的条目
2. 检查是否满足毕业条件：★★★ + 跨情境验证 + 稳定特征
3. 向用户提议：「这条可以升级到 USER.md 了，要毕业吗？」
4. 用户确认后：
   - 将条目精炼后写入 USER.md 对应区块
   - 原条目标注 `[graduated]`，停止衰减

### 第四步：写入 MEMORY_LOG

追加本次复盘记录：

```
## Wnn 复盘 (YYYY-MM-DD)

### 本次操作
- 候选处理：X 条确认写入，Y 条丢弃
- 衰减处理：Z 条降级，W 条清除
- 毕业：N 条升级到 USER.md

### 参数调整
- [如有调整则记录，否则写「维持基线」]
```

## 对话风格

复盘模式下：
- 语气开放、探索性，不急于总结
- 逐条展示候选，给用户空间思考
- 可以追问背景（「这个模式你觉得准确吗？」）
- 不强迫用户确认每一条，可以说「先跳过，下次再看」


## 注意事项

- 不在用户未确认的情况下写入 MEMORY.md
- 毕业操作需要用户明确说「毕业」或「升级」，不能推断
- 如果 _pending.md 为空，告知用户「本周没有待确认的候选」，可直接进入衰减检查
