## 快速决策树

```
调用项目B API时遇到401错误？
  ├─ 是 → 检查token是否过期（查看时间戳）
  │         ├─ 已过期 → 刷新token后重试请求
  │         └─ 未过期 → 检查其他鉴权头信息
  └─ 否 → 继续判断其他错误类型...
```

## 卡点记录

### 1. 鉴权Token过期问题

| 维度 | 内容 |
|------|------|
| **现象** | 项目B的API调用失败，返回401 Unauthorized错误，提示"token expired" |
| **根因** | 项目B的token有效期仅2小时，没有自动刷新机制，需要手动管理token生命周期 |
| **解决方案** | 1. 在调用API前检查token时间戳，提前30分钟刷新<br>2. 捕获401错误后自动重试并刷新token<br>3. 使用token管理器封装刷新逻辑 |
| **检测信号** | 看到"401 Unauthorized"或"token expired"错误信息；API调用突然失败但之前正常 |
| **记录时间** | 2026-03-25 |
| **来源** | 与项目A集成时再次遇到，属于重复出现的问题 |

## 最佳实践

### DO
- [ ] 每次API调用前检查token剩余有效期
- [ ] 实现token自动刷新机制（提前30分钟）
- [ ] 封装带重试逻辑的API客户端
- [ ] 记录token获取时间，便于追踪

### DON'T
- ❌ 假设token永久有效
- ❌ 每次失败都手动重新获取token
- ❌ 忽略401错误的具体原因
- ❌ 在项目A中硬编码项目B的token
