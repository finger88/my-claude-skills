# Theme Discovery

主题词必须可复现地从本次证据中长出来。不要把当前对话里的印象当作未标注的默认先验。

## 运行模式

先判断本次运行属于哪一种：

- `cold-start`：新 session、每日 automation、没有可靠上下文，也没有可读连续台账。主题只能从本次微信读书数据中产生。
- `continuity-start`：有可读的每日显影台账。主题仍要从本次微信读书数据中产生，但可以用台账作为 `ledger prior` 来提高连续性。
- `warm-start`：当前对话明确给了主题、书籍、上一张卡或用户追问。可以使用这些线索，但必须在证据包中标注为 `context prior`，且不能成为唯一证据。

## 数据范围

默认最小范围：

1. `/user/notebooks` 获取有笔记书籍概览。
2. 选择 1-3 本书：
   - 最近有想法的书。
   - 想法/点评密度高的书。
   - 用户当前明确提到的书。
3. 对每本书调用 `/review/list/mine`，保留用户原文想法、触发原文/划线、章节和时间。

如果只取单本书，证据水位最高只能是 `中`。如果只取少数几条想法，证据水位是 `低`。

## 从数据生成候选主题

不要从固定主题词库开始。按这个顺序生成候选：

1. **原词收集**：从用户想法中提取高频或强触发词，优先用户自己的词，不优先书中原文词。
2. **动作识别**：把静态名词改写成动作短语。例如“看见”可能变成“先成为看见的人”，“自由”可能变成“让责任不再吃掉你”。
3. **证据聚簇**：把能互相照亮的想法聚成 3-5 个候选主题簇。
4. **连续性检查**：查看台账中最近主题和开放假设，判断本次候选是在延续、转折、反证还是新增。
5. **反证检查**：如果主题只由一条想法支撑，标记为小样本；如果其他证据明显指向不同解释，要写出来。

## 候选主题评分

给每个候选主题做简短评分，不需要数字精确，但要比较：

- **证据量**：至少 2 条想法更稳，3 条以上更好。
- **覆盖度**：跨章节、跨日期、跨书籍更稳。
- **原词性**：是否来自用户原始表达，而不是代理临时概括。
- **结构性**：是否能回答“哪个我正在显影”，而不只是一个情绪标签。
- **生成性**：是否能带出下一步追问或行动。
- **源头强度**：是否有一句适合放进 `sourceEcho` 的原始想法。
- **连续性**：是否和台账中的既有主题形成推进，而不是每日断裂。

## 选择规则

优先选择：

- 有一簇证据支撑，而不是单句灵感。
- 主题词本身带动作或结构。
- 能保留用户原始表达的触发力。
- 和上一张卡不重复，除非出现了新证据推进。
- 能推进台账里的开放假设，或提出一个值得追踪的新假设。

避免选择：

- 过大的抽象词：自由、成长、关系、责任。除非能压成更具体的动作句。
- 只来自当前对话记忆、没有本次数据证据的主题。
- 只因为最近一条想法出现就选中的主题。

## 证据包必须记录

每次生成卡片前，在 `evidence.md` 写：

```markdown
## 主题词确定

- 运行模式：
- 数据范围：
- ledger prior：无 / 有，具体来自哪些旧卡
- 候选主题：
  - 主题 A：证据数 / 覆盖范围 / 为什么选或不选
  - 主题 B：证据数 / 覆盖范围 / 为什么选或不选
  - 主题 C：证据数 / 覆盖范围 / 为什么选或不选
- 选定主题：
- 选定理由：
- context prior：无 / 有，具体是什么
- 证据水位：低 / 中 / 高
```

最终回复也要简短说明主题词如何确定，让用户能看见这次显影的水源和河道。
